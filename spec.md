# Spec: create-spec-driven-app

## 1. Product context
`create-spec-driven-app` operates in the **developer tooling / specification-first delivery** domain.
It is an npm CLI that scaffolds new software projects around executable specifications rather than
code boilerplate, combining Spec-Driven Development (SDD), Domain-Driven Design (DDD),
Behavior-Driven Development (BDD) and Test-Driven Development (TDD) into a single `npx` command.

See [`PROJECT_REPORT.md`](./PROJECT_REPORT.md) for the full project overview.

## 2. Target users
- **Primary users:** Software engineers and tech leads starting a new service, module or product.
- **Secondary users:** AI coding agents (Copilot, Claude, Cursor, Aider) that consume structured specs
  to generate and review code.
- **Internal stakeholders:** Maintainers of this repository, open-source contributors, and domain-pack authors.

## 3. Main objective
Allow any team to go from zero to a fully traceable, spec-first project skeleton — with domain model,
Gherkin acceptance criteria, AI guardrails and a reproducible runtime — in under five minutes,
without prior knowledge of SDD / DDD conventions.

## 4. In scope
- The `init`, `validate` and `expand` commands and their flags.
- The `pack init` and `pack lint` subcommands for domain-pack authoring.
- The template tree (`templates/base`, `templates/backend`, `templates/frontend`, `templates/modules`).
- The domain-pack YAML schema (`pack.yaml`) and expansion engine.
- The validation quality gates enforced by `validate`.
- The runtime contract templates (Docker Compose, devcontainer, multi-environment `.env` files).
- CI/CD tooling shipped with or consumed by this repository.

## 5. Out of scope
- The content of downstream projects generated by the CLI (governed by their own generated `spec.md`).
- Hosting or managing a public domain-pack registry (tracked separately in `mejoras/`).
- IDE extensions and MCP server (Phase 3 deliverables, not yet in scope).

## 6. Success criteria
- Every CLI command has traceable Gherkin scenarios linked in `docs/specs/traceability.md`.
- The CLI can be run identically on Linux, macOS and Windows.
- A new project passes `validate` within 60 seconds of `init` on a clean machine.
- Every `pack.yaml` produced by `pack init` passes schema validation without manual edits.
- The repository's own `validate` check is green in CI on every PR.

## 7. Initial bounded contexts
| Context | Responsibility |
|---|---|
| **Scaffolding** | Parse `project.config`, render templates, materialise the project skeleton. |
| **Validation** | Enforce structural and semantic quality gates on an existing project. |
| **DomainPackExpansion** | Parse `pack.yaml`, render feature files and domain docs, enrich traceability. |
| **RuntimeConfig** | Generate Docker Compose, devcontainer and multi-environment `.env` files. |
| **Templating** | Provide a deterministic, cross-platform template engine used by all other contexts. |

## 8. Key requirements
| ID | Requirement | Priority | Status |
|---|---|---|---|
| REQ-001 | `init` generates a valid project skeleton from a `project.config` file. | Must | Verified |
| REQ-002 | `validate` enforces required files, no unresolved placeholders, traceability coverage and allowed status values. | Must | Verified |
| REQ-003 | `expand` merges a domain pack into an existing project and enriches traceability. | Must | Verified |
| REQ-004 | All commands run identically on Linux, macOS and Windows (Node ≥ 18). | Must | In Dev |
| REQ-005 | `init` generates a Docker Compose stack and multi-environment `.env` files when `DOCKER_SUPPORT=true`. | Must | Verified |
| REQ-006 | `pack init` produces a valid `pack.yaml` from interactive prompts. | Should | Draft |
| REQ-007 | `pack lint` detects orphan references, duplicate IDs and missing scenarios in a `pack.yaml`. | Should | Draft |
| REQ-008 | A GitHub Action runs `validate` on every PR in generated projects. | Should | In Dev |
| REQ-009 | Template rendering is deterministic: two identical invocations produce byte-identical output. | Must | Draft |

## 11. Domain language
| Term | Meaning |
|---|---|
| **Spec** | A structured, human-readable document that captures requirements, domain language and acceptance criteria before implementation. |
| **Pack / Domain Pack** | A YAML file (`pack.yaml`) that encodes a reusable domain model (requirements, bounded contexts, aggregates, commands, events, scenarios) for a specific business subdomain. |
| **Scaffold** | The act of generating a project skeleton from templates and configuration. |
| **Traceability matrix** | A table that links each requirement to its scenarios, domain artefacts, technical implementation and tests. |
| **Quality gate** | An automated check that a project must pass before progressing to the next phase (e.g. `validate` must exit 0). |
| **Bounded context** | A named, explicitly delimited area of the domain in which a particular model applies. |
| **Aggregate** | A cluster of domain objects treated as a single unit for data changes, with a root entity enforcing invariants. |
| **AI guardrails** | Rules encoded in `AI_RULES.md` that constrain AI-assistant contributions to the declared stack, conventions and spec artefacts. |
| **Runtime contract** | The set of environment files, Docker Compose configuration and devcontainer settings that define how the application runs across environments. |
